When standing at a decision point in a complex project, **identify the 1-2 load-bearing risks (elephants) FIRST.** Don't run small spikes, propose sub-decisions, or chase optimization questions before confirming they actually move launch math.

**Why:** Session 59 (V2 planning) — I tried to scope a 30-min I2V cost-optimization spike (Hailuo vs Kling 2.1 vs Kling Motion Control) for Pipeline B. User pushed back: *"Don't focus on small things right now. Let's address the elephant(s) in the room first if any. If not, then attack the problem priority wise."* Once I stepped back, the real elephants were obvious: pricing math (V2 COGS doubled, current credit-pack pricing → negative margin) and asset-creation pipeline (no chosen path for ~25 reference assets). The I2V cost-optimization mattered, but not before pricing was solved.

**How to apply:**
- At every "what's next?" moment, run the check: are there 1-2 risks that could *kill* this launch outright (legal, economics, capability gap, capacity)? If yes, address those before drilling into smaller decisions.
- If a spike doesn't change a launch decision, defer it to post-launch — capture in the relevant doc as a "deferred spike" with the rationale, not as an open todo.
- Before proposing a spike, ask explicitly: "is this a margin lever or a launch blocker?" Margin levers wait. Launch blockers run.
- Re-rank priorities OUT LOUD when the user signals "stop optimizing, start deciding" — they're telling you something about the framing they want, not just the work pace.
